# Translating Nextcloud Budget

Thank you for your interest in translating the Budget app! This guide explains how to contribute translations.

## Overview

The Budget app uses Nextcloud's standard translation system based on [gettext](https://www.gnu.org/software/gettext/). All user-facing strings are wrapped in translation functions, and translation files use the standard `.po` format.

**Current status:** The app is fully prepared for translations. All ~400+ user-facing strings (frontend, backend, and templates) are wrapped in translation functions. English is the default language — translations for other languages are welcome as pull requests.

## File Structure

```
budget/
├── translationfiles/
│   ├── templates/
│   │   └── budget.pot          # Source template (auto-generated, do not edit)
│   ├── de/
│   │   └── budget.po           # German translation (example)
│   ├── fr/
│   │   └── budget.po           # French translation (example)
│   └── README.md               # This file
├── l10n/
│   ├── de.json                 # Compiled German (auto-generated)
│   ├── de.js                   # Compiled German (auto-generated)
│   └── ...
└── translationtool.phar        # Nextcloud's string extraction/compilation tool
```

- `translationfiles/templates/budget.pot` — The master template containing all translatable strings. Auto-generated from source code. **Do not edit this file directly.**
- `translationfiles/<lang>/budget.po` — Translation files for each language. **This is what you edit.**
- `l10n/*.json` and `l10n/*.js` — Compiled translation files used at runtime. Auto-generated from `.po` files. **Do not edit these directly.**

## Contributing a New Translation

### Step 1: Fork and Clone

```bash
git clone https://github.com/otherworld-dev/budget.git
cd budget/budget
git checkout -b translations/<language_code>
```

### Step 2: Create Your Translation File

Copy the `.pot` template to a new language directory:

```bash
mkdir -p translationfiles/<language_code>
cp translationfiles/templates/budget.pot translationfiles/<language_code>/budget.po
```

Use [ISO 639-1 language codes](https://en.wikipedia.org/wiki/List_of_ISO_639-1_codes):

| Code | Language   | Code | Language    | Code | Language   |
|------|------------|------|-------------|------|------------|
| `de` | German     | `fr` | French      | `es` | Spanish    |
| `it` | Italian    | `pt` | Portuguese  | `nl` | Dutch      |
| `pl` | Polish     | `ru` | Russian     | `ja` | Japanese   |
| `zh` | Chinese    | `ko` | Korean      | `ar` | Arabic     |
| `sv` | Swedish    | `da` | Danish      | `nb` | Norwegian  |
| `fi` | Finnish    | `cs` | Czech       | `tr` | Turkish    |

### Step 3: Translate the Strings

Open the `.po` file in a translation editor:

- [**Poedit**](https://poedit.net/) — Recommended. Free desktop app for Windows, macOS, Linux.
- [**Lokalize**](https://apps.kde.org/lokalize/) — KDE translation editor.
- [**GTranslator**](https://wiki.gnome.org/Apps/Gtranslator) — GNOME translation editor.
- Any text editor — the `.po` format is plain text and human-readable.

#### Basic Strings

For each entry, fill in the `msgstr` with the translated text:

```po
msgid "Net Worth"
msgstr "Patrimonio Neto"

msgid "Failed to load transactions"
msgstr "Error al cargar las transacciones"

msgid "Dashboard"
msgstr "Tableau de bord"
```

#### Strings with Placeholders

Keep all placeholders exactly as they appear — they will be replaced with dynamic values at runtime:

```po
# JavaScript-style placeholders: {variableName}
msgid "Page {current} of {total}"
msgstr "Seite {current} von {total}"

msgid "Failed to save account: {error}"
msgstr "Fehler beim Speichern des Kontos: {error}"

# PHP-style positional placeholders: %1$s, %2$s
msgid "%1$s not found"
msgstr "%1$s nicht gefunden"

msgid "Invalid account type: %1$s"
msgstr "Ungültiger Kontotyp: %1$s"
```

#### Plural Forms

Some strings have singular and plural forms. The number of plural forms varies by language:

```po
# English has 2 forms (singular/plural)
msgid "%n transaction"
msgid_plural "%n transactions"
msgstr[0] "%n Transaktion"
msgstr[1] "%n Transaktionen"

msgid "%n bill"
msgid_plural "%n bills"
msgstr[0] "%n Rechnung"
msgstr[1] "%n Rechnungen"
```

For languages with more than 2 plural forms (e.g., Polish, Russian, Arabic), add the appropriate number of `msgstr[N]` entries. See [gettext plural forms](https://www.gnu.org/software/gettext/manual/html_node/Plural-forms.html) for your language's rules.

### Step 4: Update the PO Header

At the top of your `.po` file, update the header metadata:

```po
"Language: de\n"
"Plural-Forms: nplurals=2; plural=(n != 1);\n"
```

Set `Language` to your language code and `Plural-Forms` to match your language's plural rules. Poedit fills this in automatically.

### Step 5: Submit Your Translation

```bash
git add translationfiles/<language_code>/budget.po
git commit -m "i18n: Add <Language> translation"
git push origin translations/<language_code>
```

Then open a pull request on [GitHub](https://github.com/otherworld-dev/budget/pulls).

## Updating an Existing Translation

If a translation already exists but is incomplete or outdated:

1. Open `translationfiles/<language_code>/budget.po`
2. Look for untranslated entries — these have an empty `msgstr ""`
3. Look for fuzzy entries — marked with `#, fuzzy` comment, meaning the source string changed and the translation may need updating
4. Update the translations and remove the `#, fuzzy` flag when done
5. Submit a pull request with your changes

**Tip:** Poedit highlights untranslated and fuzzy entries automatically.

## Translation Tips

- **Keep placeholders intact** — `{variable}`, `%1$s`, `%n` must appear in your translation exactly as in the source
- **Match the tone** — The app uses a professional but approachable tone
- **Financial context** — Words like "Credit", "Balance", "Account" have specific financial meanings
- **Be concise** — Dashboard widget names and button labels need to fit in limited UI space
- **Don't translate** — Brand names, currency codes (EUR, USD), or technical terms that are universally understood
- **Test if possible** — Install the app with your translation to verify strings fit in the UI

## For Maintainers

### Extracting Strings (after code changes)

When new translatable strings are added to the code, regenerate the `.pot` template:

```bash
make translations
# OR manually:
php translationtool.phar create-pot-files
```

This scans all `$l->t()`, `t('budget', ...)`, `p($l->t(...))` calls and updates the template.

### Compiling Translations (after merging .po files)

After merging translation PRs, compile `.po` files to the JSON/JS format used at runtime:

```bash
make translations-compile
# OR manually:
php translationtool.phar convert-po-files
```

This generates files in the `l10n/` directory that Nextcloud loads automatically.

### How It Works at Runtime

- **Backend:** Nextcloud's `IL10N` service detects the user's language preference and loads the appropriate `.json` translation file
- **Frontend:** `@nextcloud/l10n` reads the compiled translations and the `t()` function returns the translated string (or falls back to English if no translation exists)
- **Templates:** `$l->t()` works the same as backend — translations are resolved server-side before HTML is sent
